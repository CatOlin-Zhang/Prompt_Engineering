This is an intelligent learning companion system based on Retrieval-Augmented Generation (RAG) technology. The project aims to provide students with precise academic consulting, course recommendations, and campus life guidance using a locally deployed Large Language Model (LLM). The system supports multiple document formats (PDF, DOCX, Excel, TXT) and integrates a comparison mechanism for three retrieval strategies along with an automatic LLM-based evaluation function.

---

## Key Features

- **Multi-Format Data Support**: Supports loading `.txt`, `.pdf`, `.docx`, and `.xlsx`/`.xls` files. It can intelligently process Excel table data by converting each row into an independent JSON object.
- **Three Retrieval Strategies Comparison**:
  - **Lexical Retrieval**: Keyword matching based on TF-IDF.
  - **Neural Retrieval**: Semantic vector matching based on Sentence Transformers (`all-MiniLM-L6-v2`).
  - **Hybrid Retrieval**: Min-Max normalized weighted combination of the above two methods (configurable via `HYBRID_ALPHA`), typically yielding the best results.
- **LLM Judge System**: An integrated evaluation module that automatically compares and ranks the quality of answers generated by the three strategies (Lexical / Neural / Hybrid) and outputs a ranking with justification.
- **Local Model Deployment**: Runs local models (Gemma 3) based on Ollama, ensuring data privacy and fast response.
- **Conversation Memory Management**: Supports multi-turn dialogue. The system maintains a sliding window of historical context (`MAX_HISTORY`) to understand user intent.
- **Performance Analytics**: For each query, the system reports token consumption, latency, and token efficiency for all three strategies in a comparison table.

## Environment Configuration & Installation

### 1. Install Dependencies

A `requirements.txt` file is provided in the project directory:

```bash
pip install -r requirements.txt
```

Or install the libraries manually (note: `torch` is installed automatically as a dependency of `sentence-transformers`):

```bash
pip install torch numpy pandas PyPDF2 python-docx sentence-transformers scikit-learn ollama tiktoken requests
```

### 2. Install Ollama Models

This project requires two models: one for local generation and one for evaluation (Judge). Ensure the Ollama service (`ollama serve`) is running, then pull the following models:

```bash
# Local Generation Model
ollama pull gemma3:4b

# Judge Model (used for evaluating answer quality)
ollama pull gemma3:4b-cloud
```

> If you do not have permission to call the cloud model, you can disable the LLM Judge function by entering `n` when prompted at startup.

### 3. Prepare Data

Create a folder named `data` in the project root directory and place your knowledge base files (PDF, Word, Excel, TXT) into this folder:

```text
project_root/
├── main.py
├── final-groupwork.ipynb
└── data/
    ├── academic_advising_guidelines.pdf
    ├── course_info.pdf
    └── regulations.docx
```

## How to Run

```bash
python main.py
```

On startup you will be prompted to choose a mode:

1. **Standard RAG** (default) — Runs the interactive Q&A loop directly.
2. **Advanced Demo** — Showcases advanced features (CoT, ReAct, token optimization, evaluation metrics) before entering the standard system.

You will also be asked whether to enable LLM-based comparative evaluation (`y/n`, default `y`).

### Interactive Usage

- Type your question and press Enter. The system retrieves context using all three strategies, generates a response for each, and prints a performance comparison table (tokens, latency, efficiency).
- If the LLM Judge is enabled, it ranks the three responses (e.g., `Ranking: C > A > B`) with a brief justification.
- Type `exit` or `quit` to leave the conversation.

## System Architecture & Modules

### Data Processing Module (`data_loader.py`)
- Automatically traverses the `data/` directory and calls the corresponding reader based on file extension (`.txt`, `.pdf`, `.docx`, `.xlsx`/`.xls`).
- **Smart Chunking**: For Excel/JSON data, each row is converted into an independent JSON object. The chunking logic attempts to keep object boundaries intact, avoiding splitting a single record across chunks.

### Retriever Module (`retriever.py`)
- **TF-IDF (Lexical)**: Builds a TF-IDF matrix (English stop words removed) for exact keyword matching via cosine similarity.
- **Embedding (Neural)**: Uses the `all-MiniLM-L6-v2` model to vectorize text and query, then computes cosine similarity.
- **Hybrid**: Retrieves scores from both methods over the full chunk set, applies Min-Max normalization, and combines them with a configurable weight (`HYBRID_ALPHA`, default `0.5`).

### Generator Module (`generator.py`)
- **Prompt Engineering**: Built-in prompt templates include role settings, Few-Shot examples, and Chain-of-Thought (CoT) reasoning to guide the model step-by-step. A ReAct-style tool-augmented prompt template is also provided.
- **Token Counting**: Uses `tiktoken` (`cl100k_base` encoding) to accurately count tokens of input and output.

### Judge Module (`judge.py`)
Uses a separate model (`gemma3:4b-cloud`) to compare the three responses. Evaluation criteria:
- **Accuracy**: How factually correct is the response?
- **Relevance**: How well does it address the specific question?
- **Completeness**: Does it provide sufficient information?
- **Clarity**: Is the response clear and well-structured?

The judge outputs a ranking (e.g., `Ranking: C > A > B`) with a brief justification.

### Memory Module (`memory.py`)
Maintains a sliding window of conversation history (`MAX_HISTORY` rounds) so the assistant retains context across multi-turn dialogues.

## Configuration (`config.py`)

| Parameter | Default | Description |
|---|---|---|
| `LOCAL_MODEL` | `gemma3:4b` | Ollama model used for generation |
| `JUDGE_MODEL` | `gemma3:4b-cloud` | Ollama model used for the LLM Judge |
| `OLLAMA_HOST` | `http://localhost:11434` | Ollama service address |
| `DATA_DIR` | `./data` | Knowledge base directory |
| `CHUNK_SIZE` | `300` | Target chunk size (characters) |
| `CHUNK_OVERLAP` | `50` | Chunk overlap (characters) |
| `TOP_K_CONTEXT` | `4` | Number of chunks retrieved per query |
| `TEMPERATURE` | `0.3` | Generation temperature |
| `NUM_CTX` | `4096` | Context window size |
| `HYBRID_ALPHA` | `0.5` | Lexical vs. embedding weight in hybrid retrieval |
| `MAX_HISTORY` | `3` | Conversation rounds retained in memory |

## Notes

- **Ollama Service**: Ensure the Ollama application is running in the background (`ollama serve`) before executing the code.
- **VRAM Requirements**: Running `gemma3:4b` requires sufficient GPU VRAM or system memory. If performance is slow, check your hardware resources.
- **Excel Processing**: Excel files are converted into JSON strings (one object per row) for processing. Ensure the table structure is clear to achieve the best retrieval effect.
- **Cloud Model Access**: If the `gemma3:4b-cloud` model is unavailable, disable the LLM Judge at startup to continue using the standard RAG pipeline.
